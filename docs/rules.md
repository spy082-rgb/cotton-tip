# Project Rules

## [Security & Scope Rules: 보안 및 범위 지침]
1. **작업 공간 격리**: 모든 활동은 지정된 작업 폴더(`/root`, 즉 `d:/[0] AI Google Antigravity/cotton tip`) 내부에서만 이루어져야 합니다. 그 외의 외부 폴더, C드라이브, OS 관련 시스템 파일, 파이썬 설치 폴더 등은 절대 읽거나 수정하지 마십시오.
2. **외부 통신 차단**: 사용자의 로컬 환경 설정(환경 변수 등)을 외부로 전송하거나, 검증되지 않은 외부 스크립트를 임의로 다운로드하는 행위를 엄격히 금지합니다.
3. **리소스 최적화**: 불필요한 임시 파일 생성을 최소화합니다. 압축 해제 등 부득이하게 디스크를 써야 할 경우 반드시 `/working` 내에서만 해결하고, 목적이 달성되면 즉시 파일을 삭제합니다.
4. **프로세스 종료**: 실행한 작업이나 백그라운드 프로세스가 시스템 자원을 점유하거나 파일 잠금(File Lock)을 유발하지 않도록, 작업 후 프로세스가 완전히 종료되었는지 반드시 확인합니다.

## [Workflow Rules: 작업 규칙]
1. **보고 및 승인**: 의도한 작업에 변경이 필요하거나 오류가 발생하면 즉시 멈추고 명확하게 원인을 설명한 뒤 수정 방향을 제안합니다.
2. **무한 루프 방지**: 코드를 수정하다 동일한 에러가 3번 이상 반복되면, 즉시 스스로 실행을 멈추고 현재 상황을 보고한 뒤 지시를 기다립니다.
3. **자동 백업 및 버전 관리 (매우 중요)**:
   - 프로젝트 내 중요 파일이나 코드를 크게 변경하기 전, 변경 전 원본을 `/Original` 폴더에 복사해 둡니다.
   - 파일을 삭제해야 할 경우 절대 영구 삭제하지 말고 `/trash_bin` 폴더로 이동시킵니다.
   - 의미 있는 단위의 작업 시작 전과 완료 후에는 반드시 현재 상태를 요약하여 Git 커밋(Git Commit)을 남깁니다.
4. **로그 분석**: 에러가 발생하면 무작정 코드를 바꾸지 말고, 터미널(Terminal)의 로그를 먼저 분석하여 원인을 파악합니다.
5. **구현 계획서 우선 작성**: 기능 구현 전, 전체 파일 구조와 단계별 작업 계획을 담은 `implementation_plan.md` 아티팩트를 먼저 생성하고 사용자 승인을 받습니다.
6. **테스트 및 검증**: 기능 구현 후, 정상 작동하는지 확인하는 테스트 코드를 실행하고 결과를 보고합니다.
7. **테스트 링크 의무 제공**: UI나 기능 수정 등 사용자가 시각적으로 확인해야 하는 작업을 완료한 경우, 작업 완료 보고와 함께 **반드시 사용자가 즉시 접속하여 테스트해 볼 수 있는 접속 링크를 함께 제공**해야 합니다.
